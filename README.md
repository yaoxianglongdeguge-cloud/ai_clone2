核心： 
       ai_clone  agent的初始化和运行程序所在类

       ai_des     agent工具函数描述所在类

       ai_tools   agent工具所在类
       
       这三个类初始时没有特殊特征，工具和描述都是空的。当agent实例化后，
       会读取记忆，人设，工具描述文件，工具文件，组装成一个实例agent

       每个agent的信息存在自己文件夹里。记忆和人设以文本文件形式，工具描述json，
         工具是每个工具有自己的文件夹，里面只有唯一一个++main++文件作为工具调用文件，
         读取时只会读取++main++文件
       
       每个agent，即使是管家agent与其他agent除了工具之外都是相同的，
       agent文件夹都放在根目录，文件夹的名称作为该agent的唯一标识，用于路径分割，隔离管理。

       工具是每个agent有自己的工具，不通用，每个工具由文件夹名称和++main++内函数名称标识，
       必须统一。并且除了标准库，其他的库都是动态导入。

设计思想:
       驱动和agent的信息分离，agent储存调用结构统一。

几个基本agent，框架本身自带的：
        记忆agent，用于agent的记忆概括
        工具注册agent，用于对写好的工具写描述。

接口程序：
         generate.py  用于创建/删除agent（在根目录调用此接口）

         desmake.py   用于把工具描述注册进描述文件夹，也可以修改，删除(在agent的tools文件夹调用,
                                当放好新函数文件夹，调用，注册工具)

         memory.py    用于调用记忆agent（在agent的主目录调用）

         port文件夹    大部分接口主程序实现地方

启动程序:
         main.py     作为程序的入口，当运行时会先预加载（目前还没进行实现）所有agent

根目录文件:
      
      agent_login:当用generate.py接口创建agent后会在这里注册，当启动main后会预加载所有agent

      agent_call_queue:中心agent需要规划好流程发送到这里，就可以按流程执行

      call_structure:规划好的流程要按照这个格式写
       
       
那么流程就是generate.py创建agent，到tools文件夹，放好工具函数文件夹（文件夹名称和++main++里函数名称必须相同）注册，
之后就可以调用了。人设的话在role.txt文件自己写就行，我认为没必要专门写个agent。


tools制作规定:
     1.除标准库外，其余导入均需要再函数内部。
     2.工具不能修改工具文件夹外文件，但能读。


通信机制：
     类似于计算机网络通信，首先每个agent有自己的收件箱和发件箱，所有agent有一个共享消息队列，agent的每条消息必须要发出去，必须有一个发送对象。
     agent返回结构化文本，代码解析并打包之后放入发件箱，之后放入共享队列，接线员把共享队列的包解析，转发给对应的agent，也可以是所有agent或者user。
     agent，接线员都是并发持续扫描的。
